در رباتیک، «روی سیستم من کار می‌کند» معمولا آغاز یک دردسر طولانی است. یک پروژه ممکن است به نسخه مشخصی از Ubuntu، ROS 2، Python، Gazebo، CUDA، درایور دوربین، کتابخانه‌های پردازش تصویر و حتی تنظیمات شبکه وابسته باشد. کافی است یکی از این قطعات روی لپ‌تاپ عضو دیگر تیم متفاوت نصب شده باشد تا همان کدی که دیروز در شبیه‌ساز اجرا می‌شد، امروز با خطاهای عجیب متوقف شود. Docker دقیقا برای همین جنس مسئله ارزشمند است: محیط اجرای پروژه را همراه خود پروژه بسته‌بندی می‌کند تا توسعه، تست و انتقال بین سیستم‌ها قابل تکرارتر شود.

این مقاله قرار نیست Docker را به عنوان یک ابزار صرفا DevOps معرفی کند. در رباتیک، Docker یک ابزار عملی برای آرام‌تر کردن آشفتگی وابستگی‌هاست؛ مخصوصا وقتی ROS 2، Gazebo، Isaac ROS، GPU، سنسورها و چندین سرویس همزمان وارد بازی می‌شوند.

> وقتی محیط توسعه قابل تکرار باشد، انرژی تیم به جای رفع خطای نصب، صرف طراحی nodeها، شبیه‌سازی سناریوها و بهتر کردن رفتار ربات می‌شود.

## Docker چیست و چرا برای رباتیک مهم است؟

Docker نرم‌افزار را داخل محیط‌های ایزوله‌ای به نام container اجرا می‌کند. هر container از روی یک image ساخته می‌شود؛ image را می‌توان مثل یک نسخه آماده از محیط پروژه دید که سیستم‌عامل پایه، ابزارها، کتابخانه‌ها و وابستگی‌های لازم را در خود دارد. تفاوت مهم اینجاست که به جای نصب دستی ROS 2، Gazebo و ده‌ها پکیج روی سیستم هر نفر، یک تصویر مشترک ساخته می‌شود و همه اعضای تیم همان را اجرا می‌کنند.

فرض کنید تیمی روی ROS 2 Humble کار می‌کند. یک نفر Ubuntu 22.04 دارد، نفر دیگر روی Windows با WSL توسعه می‌دهد و نفر سوم می‌خواهد بخشی از سیستم را روی Jetson تست کند. اگر همه چیز دستی نصب شود، اختلاف نسخه‌ها به سرعت خودش را نشان می‌دهد؛ اما با Docker می‌توان یک محیط مشترک ساخت که workspace، dependencyها، ابزارهای build، نسخه ROS و حتی ابزارهای شبیه‌سازی در آن مشخص شده‌اند. نتیجه این است که پروژه به جای وابسته بودن به حافظه و تنظیمات شخصی اعضای تیم، به یک فایل قابل بازبینی و قابل اشتراک‌گذاری وابسته می‌شود.

اگر می‌خواهید تصویر کلی را با یک توضیح ویدیویی ببینید، این ویدیو می‌تواند نقطه شروع خوبی باشد: [Docker for Robotics](https://www.youtube.com/watch?v=SAMPOK_lazw). بعد از دیدن آن، بخش‌های بعدی مقاله روشن‌تر می‌شوند، چون می‌بینید مشکل اصلی فقط نصب پکیج نیست؛ مسئله ساختن محیطی است که بارها و روی ماشین‌های مختلف قابل اجرا باشد.

## از ROS تا Gazebo؛ Docker چه مشکلی را حل می‌کند؟

ROS 2 معمولا از چندین node تشکیل می‌شود که با topic، service، action و DDS با هم ارتباط می‌گیرند. در کنار آن، Gazebo یا ابزارهای visualization مثل RViz هم وارد مسیر می‌شوند. حالا اگر perception، navigation، simulation و logging هرکدام dependencyهای متفاوتی داشته باشند، نگه داشتن یک سیستم تمیز و پایدار سخت می‌شود. Docker کمک می‌کند این بخش‌ها را به شکل روشن‌تری جدا کنیم و برای هر بخش محیط مناسب خودش را بسازیم.

در یک سناریوی ساده، می‌توان یک container برای workspace اصلی ROS 2 داشت؛ کد پروژه با volume از سیستم میزبان به container وصل می‌شود، اما build و run داخل container انجام می‌گیرد. در سناریوهای حرفه‌ای‌تر، Docker Compose وارد می‌شود و چند container را همزمان بالا می‌آورد: یکی برای simulation، یکی برای perception، یکی برای navigation و یکی برای ابزارهای جانبی. این الگو باعث می‌شود معماری نرم‌افزار به معماری واقعی ربات نزدیک‌تر شود، چون هر بخش مرز و مسئولیت مشخص‌تری پیدا می‌کند.

> Docker جای فهمیدن ROS 2، شبکه و لینوکس را نمی‌گیرد؛ اما کاری می‌کند این مفاهیم در یک محیط قابل کنترل تمرین و اجرا شوند.

## Gazebo و ابزارهای گرافیکی داخل container

یکی از تفاوت‌های مهم رباتیک با بسیاری از پروژه‌های نرم‌افزاری این است که معمولا فقط با terminal سروکار نداریم. برای توسعه ربات باید شبیه‌ساز، visualization، دوربین مجازی، lidar، نقشه و حرکت ربات را ببینیم. اجرای ابزارهایی مثل Gazebo و RViz داخل container ممکن است، اما به تنظیمات بیشتری نیاز دارد؛ مثلا دسترسی به display سیستم میزبان از طریق X11 یا Wayland، تنظیم permissionها، و در بعضی حالت‌ها استفاده از VNC یا راهکارهای remote visualization.

برای پروژه‌های سبک‌تر، اجرای GUI از طریق X11 روی لینوکس کافی است. برای پروژه‌های سنگین‌تر، مخصوصا زمانی که GPU یا simulation سه‌بعدی جدی داریم، باید دسترسی container به GPU هم درست تنظیم شود. اینجاست که تفاوت بین «Docker را نصب کرده‌ام» و «Docker را برای رباتیک درست استفاده می‌کنم» خودش را نشان می‌دهد.

## Isaac ROS و مسیر حرفه‌ای‌تر توسعه

در پروژه‌هایی که پردازش تصویر، شتاب‌دهی GPU، depth camera یا inference بلادرنگ جدی است، اکوسیستم NVIDIA Isaac ROS نمونه خوبی از یک workflow کانتینری است. در این سبک توسعه، محیط build و ابزارها داخل container نگه داشته می‌شوند تا workspace در شرایط consistent و reproducible ساخته شود. این کار به‌خصوص برای تیم‌هایی مهم است که بین شبیه‌سازی، تست روی workstation و اجرای نهایی روی Jetson جابه‌جا می‌شوند.

برای تکمیل این بخش، این ویدیو هم می‌تواند نگاه عملی‌تری بدهد: [Docker and Robotics Workflow](https://www.youtube.com/watch?v=kxAF-Pye8rI). بهتر است آن را بعد از آشنایی اولیه با مفهوم image و container ببینید، چون تمرکز اصلی در این مرحله روی workflow توسعه و انتقال محیط است.

## از کجا شروع کنیم؟

برای شروع، لازم نیست همان ابتدا یک معماری پیچیده چند-container بسازید. مسیر منطقی این است که اول یک image آماده ROS 2 را اجرا کنید، بعد workspace خودتان را با volume به container وصل کنید، سپس یک پکیج ساده را build و run بگیرید. وقتی این مرحله پایدار شد، اجرای Gazebo یا RViz را اضافه کنید و بعد سراغ Dockerfile اختصاصی پروژه بروید. Dockerfile به شما اجازه می‌دهد نصب dependencyها و تنظیمات محیط را به جای یک راهنمای دستی طولانی، در قالب دستورهای روشن و قابل بازبینی نگه دارید.

در مرحله بعد می‌توانید Docker Compose را وارد کنید. مثلا simulation، navigation و perception را در سرویس‌های جدا بالا بیاورید و ارتباط آن‌ها را از طریق شبکه Docker و تنظیمات ROS 2 بررسی کنید. این مرحله برای یادگیری معماری سیستم هم عالی است، چون خیلی سریع مشخص می‌شود کدام node به کدام سرویس، topic، sensor یا permission وابسته است.

> بهترین تمرین این است که یک پروژه کوچک ROS 2 را انتخاب کنید، یک Dockerfile برای آن بسازید و از خودتان بپرسید: آیا عضو دیگری از تیم می‌تواند فقط با clone کردن repo و اجرای یک دستور، همان محیط را بالا بیاورد؟

## اشتباهات رایج هنگام استفاده از Docker در رباتیک

اولین اشتباه، فراموش کردن GPU است. اگر پروژه شما Gazebo سنگین، پردازش تصویر، Isaac ROS یا inference دارد، container باید به GPU دسترسی داشته باشد و نسخه‌های CUDA، driver و runtime با هم سازگار باشند. بدون این هماهنگی، ممکن است کد درست باشد اما اجرا کند، ناپایدار یا کاملا ناموفق شود.

دومین اشتباه، بی‌توجهی به سخت‌افزار واقعی است. روی ربات واقعی، container ممکن است به camera، LiDAR، joystick، serial port، USB یا شبکه داخلی ربات نیاز داشته باشد. پس باید از همان ابتدا به permissionها، device mapping و امنیت دسترسی‌ها فکر کرد. Docker می‌تواند محیط را تمیزتر کند، اما اگر بدون دقت به سخت‌افزار وصل شود، عیب‌یابی را سخت‌تر می‌کند.

سومین اشتباه، ساده گرفتن شبکه در ROS 2 است. ROS 2 از DDS استفاده می‌کند و discovery بین nodeها به تنظیمات شبکه حساس است. وقتی nodeها در containerهای جدا یا روی ماشین‌های مختلف اجرا می‌شوند، باید network mode، domain id، multicast و interfaceهای شبکه را با دقت بررسی کرد. خیلی وقت‌ها مشکل از کد نیست؛ nodeها فقط همدیگر را پیدا نمی‌کنند.

## جمع‌بندی

Docker برای توسعه‌دهنده رباتیک یک میانبر جادویی نیست، اما یکی از بهترین ابزارها برای ساختن محیط‌های قابل تکرار، قابل اشتراک‌گذاری و نزدیک‌تر به deployment واقعی است. اگر با ROS 2، Gazebo یا Isaac ROS کار می‌کنید، یاد گرفتن Docker باعث می‌شود پروژه‌هایتان کمتر به تنظیمات اتفاقی سیستم‌ها وابسته باشند و تیم بتواند با اعتماد بیشتری بین توسعه، شبیه‌سازی و اجرای واقعی حرکت کند.

از یک container ساده شروع کنید، بعد workspace را mount کنید، سپس GUI و GPU را اضافه کنید و در نهایت به Docker Compose برسید. این مسیر قدم‌به‌قدم هم برای یادگیری Docker مناسب است و هم برای فهمیدن معماری واقعی سیستم‌های رباتیکی.
